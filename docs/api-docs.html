HealthcareService

Canonical catalog services and instance subscription services.

Search services

Search canonical catalog services and organization-specific instance subscriptions. Identifier searches can target either PCM HealthcareService identifier system: http://pcm.fhir.health.gov.il/identifier/pcm-healthcareservice-catalog-id for canonical catalog entries, or http://pcm.fhir.health.gov.il/identifier/pcm-healthcareservice-instance-id for organization-specific instance subscriptions.

Default visibility is active-biased. When the active query parameter is omitted, PCM returns active catalog entries and active instance services. The authenticated caller also sees its own instance services even when active=false, so an organization can inspect subscriptions that it disabled and can request reactivation when server-side rules permit it.

Explicit active=false searches are authorization-limited. Ordinary organizations can see only their own inactive instance services. PCM administration can inspect inactive catalog and instance records for operational management.

Register service or request a new service

Service providers discover active catalog services first with GET /HealthcareService?active=true. If a suitable active canonical catalog service exists, the provider posts a minimal instance HealthcareService with the basedOn extension to register for that service. If no suitable service exists, the provider posts a full new-service registration request through the same instance flow, including service name, purpose, information buckets, history depth, access mode, access duration, clinical information, and example queries.

Submit profiles are illustrative implementation guidance only and are not sent in meta.profile. Runtime service submit requests are distinguished by meta.tag=instance and payload shape: basedOn present means existing catalog registration; no basedOn with a full service definition means a new-service registration request. Created/read response examples use the returned pcm-healthcareservice-instance profile.

External clients do not directly create an immediately active canonical catalog service. For a valid new-service registration request, PCM creates both the canonical catalog service and the provider's organization registration with active=false. Activation and any required review or exception handling are managed outside this FHIR operation. Clients observe the result through HealthcareService.active and OperationOutcome responses, not through separate public review resources.

Full service definitions require a human-readable service purpose. The public FHIR representation for that purpose is HealthcareService.extraDetails, carrying the service description/purpose ("תיאור השירות (מטרה)"). Client-supplied active values are controlled by PCM and may be ignored or overridden. PCM assigns the stable instance/subscription business identifier using system http://pcm.fhir.health.gov.il/identifier/pcm-healthcareservice-instance-id; clients should omit identifier on instance create/update. Clients should include meta.tag=instance on service POST/PUT requests so request intent is explicit, but should not include meta.profile on submit request payloads.

Update service instance state or submit catalog update request

Instance owners may submit a minimal PUT containing active=false to disable their own instance subscription. They may submit active=true to request reactivation, but PCM allows that transition only when server-side rules permit external use; otherwise it returns an OperationOutcome and leaves the resource inactive. Non-writable elements are ignored or overwritten by PCM. Clients should include meta.tag=instance on service PUT requests, but should not include meta.profile on submit/update request payloads.

Catalog update requests may also be submitted through this operation. PCM does not mutate the active canonical catalog entry inline. Changes to information buckets, patient-consent access parameters, tagged clinical data, or history depth are rejected with an OperationOutcome and should be submitted through the new-service registration flow. Changes accepted for asynchronous processing return a 202 Accepted OperationOutcome, and the active catalog entry remains unchanged until PCM applies the update.

Create consent request

Create a proposed consent request for a patient and a service provider's HealthcareService instance. The pcm-service extension must reference an existing active instance service. PCM rejects requests that omit the service, point to a canonical catalog service, or point to an inactive or otherwise unusable instance. Token issuance later repeats this service-state check, so an existing active Consent does not authorize data-source access after the service instance becomes inactive. The accepted request shape is documented by the illustrative pcm-consent-submit profile, but clients do not send that profile in meta.profile. scope=patient-privacy and category=INFA are mandatory fixed values on submit. Created/read response examples use the returned pcm-consent profile.

Issue access token (client_credentials)

Issues an opaque access token using client_credentials OAuth2 flow. The endpoint requires mTLS as a transport-level policy, while OAuth2 client authentication is performed with private_key_jwt. The issued token is associated with the registered OAuth client identity and the private key used to sign the client assertion. If the presented mTLS certificate differs from the JWT-signing certificate/key identity, the assertion should include cnf certificate evidence according to the configured PCM policy. For data source access, the client_assertion MUST include UDAP B2B extensions with a Consent reference (URL). PCM issues each access token for exactly one resource server. The token request MUST contain exactly one RFC 8707 resource parameter; requests containing more than one resource parameter fail with invalid_target. Clients request one token per resource server.
